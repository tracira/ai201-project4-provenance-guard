## Architecture

### Narrative

A piece of text follows this path from submission to the label a user sees:

1. **Rate Limiter** — the request hits `POST /submit` first. Flask-Limiter checks the submitting IP against the configured limits. If exceeded, the request is rejected with 429 before any detection work begins.

2. **Signal 1 — LLM Classifier (Groq)** — the text is sent to `llama-3.3-70b-versatile` with a structured prompt that asks the model to return a JSON object with an `ai_probability` float (0.0–1.0) and a one-sentence rationale. This becomes `llm_score`. It captures semantic and holistic coherence: voice, specificity, tonal consistency, and the kind of "averaging over training data" structure that AI prose tends to produce.

3. **Signal 2 — Stylometric Heuristics (pure Python)** — four statistical properties are computed from the raw text. Each is normalized to a 0–1 sub-score (0 = human-like, 1 = AI-like) and averaged into `stylo_score`. The four sub-signals are sentence length variance, type-token ratio, expressive-punctuation diversity, and average word length. See Spec § Signal Design for exact formulas.

4. **Confidence Scorer** — `final_score = 0.65 × llm_score + 0.35 × stylo_score`. The result maps to a classification (`human`, `ai_generated`, `uncertain`) and a confidence level (`high` or `uncertain`). See thresholds in Spec § Uncertainty Representation.

5. **Transparency Label Generator** — `final_score` and `classification` are passed to a label function that returns one of three plain-English strings (see Spec § Transparency Label Variants).

6. **Audit Log (SQLite)** — every decision is written to the `audit_log` table before the response is returned: content ID, both signal scores, final score, classification, label text, status. Appeals are appended to the same row.

7. **HTTP Response** — structured JSON including `content_id`, `classification`, `confidence_score`, `label_text`, `signals`, `status`, and `appeal_url`.

## Spec

### 1. Signal Design

#### Signal 1: LLM Classifier

**What it measures:** holistic semantic and stylistic coherence. Does the text feel like a human with opinions, experience, and imprecision — or does it read like a model averaging over training data?

**Why it differs between human and AI writing:** AI text trends toward confident completeness, smooth transitions, and "covering all bases" structure. Human writing is more idiosyncratic: incomplete thoughts, tonal shifts, digressions, and specificity rooted in lived experience rather than statistical plausibility.

**Prompt format (exact):**
```
You are an expert at distinguishing human-written text from AI-generated text.

Analyze the following text and estimate the probability that it was generated by AI.

Return ONLY a valid JSON object with exactly these two fields:
- "ai_probability": a float from 0.0 to 1.0
    (0.0 = definitely human-written, 1.0 = definitely AI-generated)
- "reasoning": a single sentence explaining your assessment

Text:
"""
{content}
"""
```

**Output format:** `{"ai_probability": 0.73, "reasoning": "The text exhibits..."}`

**How `llm_score` is extracted:** parse the JSON response and take `ai_probability` as a float. On any parse failure (malformed JSON, missing field, out-of-range value), default to `0.5` and flag `llm_parse_error: true` in the response.

**Edge handling:** if the text is fewer than 15 words, still call the LLM but the response is noted as potentially unreliable (flagged in the audit log, but score is used as-is).

**Blind spots:**
- AI text deliberately prompted with typos, informal registers, or contradictions can fool this signal.
- Well-edited polished human prose (professional essays, revised short stories) may be scored as AI-like.
- The detector and the content share training biases — the model may not recognize its own output patterns reliably.

---

#### Signal 2: Stylometric Heuristics

**What it measures:** four statistical surface properties that differ between human and AI writing.

**Why it differs:** AI text is generated token-by-token to maximize local coherence, producing statistically smoother output. Human writing has higher variance in all four properties.

**Sub-signal 1: Sentence Length Variance**

Measures how much sentence length varies across the text. AI prose tends toward uniform sentence lengths; human writing oscillates.

```python
import re, statistics

def sentence_length_score(text):
    sentences = [s.strip() for s in re.split(r'[.!?]+', text) if s.strip()]
    if len(sentences) < 2:
        return 0.5  # insufficient data
    lengths = [len(s.split()) for s in sentences]
    mean = statistics.mean(lengths)
    if mean == 0:
        return 0.5
    cv = statistics.stdev(lengths) / mean  # coefficient of variation
    # cv near 0 = uniform (AI-like) → score 1.0
    # cv above 0.8 = highly variable (human-like) → score 0.0
    return max(0.0, min(1.0, 1.0 - (cv / 0.8)))
```

**Sub-signal 2: Type-Token Ratio (TTR)**

Measures vocabulary diversity: unique words / total words. AI tends to use a slightly narrower effective vocabulary in a given passage. Note: TTR is length-dependent — very short texts have naturally high TTR regardless of authorship. For texts under 30 words, this sub-signal returns `0.5` (neutral).

```python
def ttr_score(text):
    words = re.findall(r'\b\w+\b', text.lower())
    if len(words) < 30:
        return 0.5
    ttr = len(set(words)) / len(words)
    # ttr near 0.7+ = diverse vocabulary (human-like) → score 0.0
    # ttr near 0.4  = less diverse (AI-like) → score 1.0
    return max(0.0, min(1.0, (0.65 - ttr) / 0.25))
```

**Sub-signal 3: Expressive Punctuation Diversity**

Counts how many distinct "expressive" punctuation characters appear. Human writing uses em-dashes, semicolons, ellipses, and exclamation marks more variably; AI prose tends toward periods and commas only.

```python
def punctuation_score(text):
    expressive = set('—–;…!?')
    found = {c for c in text if c in expressive}
    count = len(found)
    # 0 types → 1.0 (AI-like, punctuation-poor)
    # 3+ types → 0.0 (human-like, punctuation-rich)
    return max(0.0, 1.0 - count / 3.0)
```

**Sub-signal 4: Average Word Length**

AI text trends toward slightly longer, more formal words. Human conversational and creative writing tends toward shorter words.

```python
def word_length_score(text):
    words = re.findall(r'\b\w+\b', text)
    if not words:
        return 0.5
    avg_len = sum(len(w) for w in words) / len(words)
    # avg 4.0 chars → 0.0 (short words, human-like)
    # avg 6.0 chars → 1.0 (long words, AI-like)
    return max(0.0, min(1.0, (avg_len - 4.0) / 2.0))
```

**Combined `stylo_score`:**

```python
def compute_stylo_score(text):
    sub_scores = [
        sentence_length_score(text),
        ttr_score(text),
        punctuation_score(text),
        word_length_score(text),
    ]
    return sum(sub_scores) / len(sub_scores)
```

**Blind spots:**
- Poetry with intentional repetition (anaphora) has low TTR and uniform sentence structure by design — this signal will falsely lean AI for poetry. See Edge Cases.
- Very short texts (< 30 words) produce unreliable statistics; TTR sub-signal is suppressed, others return neutral.
- Heavily edited AI drafts inherit human variance introduced during revision — this signal will under-flag them.

---

#### Combining Signals into `final_score`

```python
LLM_WEIGHT   = 0.65
STYLO_WEIGHT = 0.35

def compute_final_score(llm_score, stylo_score):
    return LLM_WEIGHT * llm_score + STYLO_WEIGHT * stylo_score
```

**Rationale for 0.65/0.35:** the LLM signal captures semantic and holistic properties (voice, naturalness, coherence) that surface statistics cannot. Stylometrics is a useful corroborating signal but is more easily fooled by text type. The LLM signal is also more robust for very short texts where statistics are unreliable. The 0.35 weight on stylometrics is enough to meaningfully shift the score when the signals agree but not enough to override the LLM when they disagree.

### 2. Uncertainty Representation

**Thresholds:**

| final_score range | classification | confidence_level |
|-------------------|---------------|-----------------|
| < 0.35            | `human`        | `high`           |
| 0.35 – 0.65       | `uncertain`    | `uncertain`      |
| > 0.65            | `ai_generated` | `high`           |

**What a score of 0.60 means:** it is in the uncertain band. It means the combined weighted signal leans slightly toward AI (above 0.5) but not enough to classify with confidence. A typical way this arises: LLM returns `0.72` (leans AI) and stylometrics return `0.38` (leans human), giving `0.65×0.72 + 0.35×0.38 = 0.468 + 0.133 = 0.601`. The signals are in partial disagreement and neither is overwhelming — the system honestly reports it cannot decide.

**What a score of 0.51 vs 0.95 means:** both map to the same `classification` (`ai_generated` is at `> 0.65` — wait: 0.51 is in the *uncertain* band). More concretely:
- `0.51` → `uncertain` → hedged label that invites appeal
- `0.95` → `ai_generated`, high confidence → direct AI label with appeal option

The numeric score is returned in the response alongside the label, so a developer downstream can apply their own threshold if needed. The label communicates the meaningful difference to end users.

**Why a wide uncertain band (30 points wide):** it reflects the known asymmetry — a false positive (labeling a human's work as AI-generated) harms a creator in a concrete way (reputational damage, platform penalties). A false negative (missing AI content) is a lesser platform failure. The system is designed to err toward "I don't know" rather than confident accusation. The 35-point human-confidence zone is smaller than the 35-point AI-confidence zone — we require slightly more evidence to call something "confidently human" because human writing has more variance and edge cases.

**How calibration is tested:** submit a reference set of 5 known-human texts (published creative work with attributable human authors) and 5 known-AI texts (outputs from GPT-4o/Claude on the same prompts). Expected: AI texts cluster above 0.65 and human texts cluster below 0.5. If the distributions fully overlap, the scoring is not meaningful. Record these reference scores in the README as evidence.

### 3. Transparency Label Variants

These are the exact strings the `generate_label(final_score)` function returns. They are shown verbatim to the end user as the transparency label.

**High-confidence AI** (`final_score > 0.65`):

> "Our automated analysis found strong signals that this content may be AI-generated. This is not a definitive finding — our system can be wrong, especially for polished or formally written human work. If you are the creator and believe this assessment is incorrect, you can submit an appeal."

**Uncertain** (`0.35 ≤ final_score ≤ 0.65`):

> "Our system could not determine with confidence whether this content was written by a human or generated by AI. The signals were mixed or inconclusive. If you are the creator, you may submit an appeal to provide additional context that can help with review."

**High-confidence human** (`final_score < 0.35`):

> "Our automated analysis found strong signals that this content was written by a human. This assessment is based on stylistic and semantic patterns and reflects our best confidence at this time."

**Implementation:**

```python
def generate_label(final_score):
    if final_score > 0.65:
        return (
            "Our automated analysis found strong signals that this content "
            "may be AI-generated. This is not a definitive finding — our system "
            "can be wrong, especially for polished or formally written human work. "
            "If you are the creator and believe this assessment is incorrect, "
            "you can submit an appeal."
        )
    elif final_score < 0.35:
        return (
            "Our automated analysis found strong signals that this content was "
            "written by a human. This assessment is based on stylistic and semantic "
            "patterns and reflects our best confidence at this time."
        )
    else:
        return (
            "Our system could not determine with confidence whether this content "
            "was written by a human or generated by AI. The signals were mixed or "
            "inconclusive. If you are the creator, you may submit an appeal to "
            "provide additional context that can help with review."
        )
```

### 4. Appeals Workflow

**Who can submit an appeal:** anyone who has the `content_id` from a prior `POST /submit` response. No authentication is required in the MVP — the creator is assumed to be the person who made the original submission and retained the `content_id`. A production system would require auth; this is documented in the README as a known limitation.

**What the creator provides:**
- `content_id` (in the URL path)
- `creator_id` (string, self-reported identifier — not validated)
- `reasoning` (free-text explanation of why they believe the classification is wrong)

**What happens when an appeal is received:**
1. Look up `content_id` in the `audit_log` table
2. If not found: return `404 {"error": "content_id not found"}`
3. If `status` is already `under_review`: return `409 {"error": "appeal already submitted for this content"}`
4. Write `appeal_reasoning` and `appeal_timestamp` to the existing audit log row
5. Update `status` to `under_review`
6. Return `200` with `{"appeal_id": uuid, "content_id": ..., "status": "under_review", "message": "Your appeal has been received and the content is now under review."}`

**What a human reviewer sees (via `GET /log`):**

Each log entry in the appeal queue exposes:
- `content_preview`: first 200 chars of the original text
- `llm_score` + `stylo_score`: raw signal values (reviewer can judge whether the signals were borderline)
- `final_score`: the weighted combined score
- `classification` + `confidence_level`: the decision that was made
- `label_text`: the exact label the creator saw
- `appeal_reasoning`: the creator's explanation in their own words
- `appeal_timestamp`: when the appeal was submitted
- `status`: `under_review`

This gives a reviewer everything needed to make a manual decision without needing to re-run the system: the original signals, the decision, and the creator's rebuttal.

### 5. Anticipated Edge Cases

**Edge Case 1: Anaphoric or repetitive poetry**

*Scenario:* A creator submits a poem with heavy structural repetition — e.g., each stanza begins with "I am the…", uses simple vocabulary, and keeps line lengths uniform. This is a deliberate literary device (anaphora).

*What the system does wrong:* sentence length variance will be low → `sentence_length_score` near 1.0. TTR will be low due to repeated phrases → `ttr_score` near 1.0. Expressive punctuation may be minimal in spare poetry → `punct_score` near 1.0. The stylometric signal will strongly flag this as AI. The LLM may also be confused if the poem lacks the idiosyncratic voice cues it looks for. Expected result: false positive, high AI score.

*Mitigation:* the label for high-confidence AI explicitly acknowledges "our system can be wrong, especially for polished or formally written human work." The appeal pathway is the safety valve. This edge case should be documented in the README as a known limitation.

**Edge Case 2: Very short texts (under 30 words)**

*Scenario:* A creator submits a haiku, a tagline, or a short dedication (e.g., five lines of a poem). Sentence length variance is computed from 1–3 data points — statistically unreliable. TTR is suppressed (returns 0.5). The LLM may produce a confident score on thin evidence.

*What the system does wrong:* the LLM may still return a low or high `ai_probability` with apparent confidence because it's pattern-matching on surface features. The stylometric signal is partially suppressed but not wholly. The `final_score` may be misleading.

*Mitigation:* the system flags texts under 30 words in the response with `"short_text_warning": true` and adds a note to the audit log. The label for uncertain scores already communicates "signals were mixed or inconclusive," which is the honest description. Extremely short texts (<10 words) return `final_score = 0.5` with classification `uncertain` without calling the LLM (noted in README).

**Edge Case 3: Non-English text**

*Scenario:* A German-language poem is submitted. The LLM classifier can still assess it but its calibration was likely developed more on English text. The stylometric sub-signal for average word length will be inflated (German compound words are long) and will falsely skew the score toward AI-like.

*Mitigation:* not handled in the MVP. The README documents that the system is calibrated for English-language content. Non-English content should expect degraded accuracy.

## Rate Limiting

**Limits:**
- 5 requests per minute per IP
- 50 requests per hour per IP

**Reasoning:** A real creative writer submits finished work — a poem, a story, a blog post. Submitting more than 5 pieces per minute is implausible for legitimate use. An adversary probing the classifier or exhausting the Groq API quota would need to submit at volume; 5/min forces any flood to take at minimum 12× longer. The hourly cap of 50 backstops slow-drip attacks that stay under the per-minute limit. Both limits are per-IP to avoid blocking other users when one IP hits the cap. These limits are intentionally conservative in the MVP; a production deployment would tune them against actual usage data.

## Audit Log Schema

```sql
CREATE TABLE IF NOT EXISTS audit_log (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    content_id       TEXT NOT NULL UNIQUE,
    creator_id       TEXT,
    timestamp        TEXT NOT NULL,
    content_preview  TEXT,
    llm_score        REAL,
    stylo_score      REAL,
    final_score      REAL,
    classification   TEXT,
    confidence_level TEXT,
    label_text       TEXT,
    short_text_flag  INTEGER DEFAULT 0,
    status           TEXT NOT NULL DEFAULT 'classified',
    appeal_reasoning TEXT,
    appeal_timestamp TEXT
);
```

Every `POST /submit` inserts one row. `POST /appeal/<content_id>` updates the existing row (it does not insert a new one — appeals are part of the same record as the original decision). `GET /log` reads this table ordered by `id DESC`.
